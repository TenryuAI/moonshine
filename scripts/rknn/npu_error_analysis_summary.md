# Silero VAD NPU 数值偏差根因排查总结

## 1. 排查过程与发现

通过“控制变量法”和“逐层 Dump”技术，我们对 CPU 和 RK3568 NPU 运行 `silero_vad_k2` 模型的数值偏差进行了根本原因分析。

### 1.1 FP16 纯数学精度基线验证 (Host 侧)
我们首先怀疑是 FP16 数据类型的尾数截断导致了 LSTM 状态的累积误差。
为此，我们在主机侧使用 Python/NumPy 将模型的输入和状态强制转换为 FP16，模拟纯 FP16 精度下的推理。
**结果**：
- 纯 FP16 模拟的 `prob_max_abs` 仅为 `0.002650`。
- `voice_mismatch_windows` 为 `0`。
- 结论：**纯 FP16 精度完全足以支撑该模型的稳定运行，误差并非单纯由 FP16 数据类型引起。**

### 1.2 单窗口全节点输出对比 (Device 侧)
排除了纯数据类型原因后，我们将 `backend.onnx` 的所有 52 个中间节点暴露为输出，转换为 RKNN 模型，并在 RK3568 上逐窗比对 CPU 和 NPU 的中间张量。
**结果**：
- 在第 1 个时间窗（Window 0），绝大多数 Conv 和 Add 算子的最大误差（MaxDiff）在 `0.002 - 0.004` 之间，表现正常且稳定。
- 然而，LSTM 的 Cell State (`new_c` / `LSTM_output_2`) 误差从 Window 0 的 `0.0015` 开始，以每窗口约 `0.004` 的速度**线性递增**。
- 到 Window 9 时，`new_c` 的 MaxDiff 已经放大到 `0.0365`。
- 结论：**误差的根源在于 RK3568 NPU 对 LSTM 算子（或其内部的 Sigmoid/Tanh 激活函数）的硬件/底层驱动实现存在微小偏差。由于 LSTM 的自回归特性，这个微小偏差在 $C_t$ 状态中被不断累积放大，最终导致 VAD 概率输出翻转。**

## 2. 根本原因结论

**RK3568 NPU 的 LSTM 算子实现精度不足，导致递归状态（Cell State）误差随时间步不断累积，最终破坏了模型的时序语义。**
这通常是因为 NPU 在实现非线性函数（Sigmoid/Tanh）时使用了低精度的查找表（LUT）或近似算法，其误差超出了标准 IEEE FP16 的误差范围。

## 3. 修复方案与建议

鉴于根因在 NPU 硬件/驱动底层，应用层的修复建议如下：

### 方案一：异构切图调度 (最佳推荐)
既然 NPU 的卷积运算精度正常且算力强，而 LSTM 运算存在累积误差且算力需求低，最佳方案是**将模型再次切分**：
1. **NPU 运行 Encoder (Conv 层)**：提取音频特征，发挥 NPU 算力优势。
2. **CPU 运行 Decoder (LSTM 层)**：接收 NPU 的特征，在 CPU 上进行高精度的 LSTM 状态更新，彻底消除累积误差。
*注：LSTM (单步长, hidden=64) 在 CPU 上的计算开销极小，这种异构调度能达到性能与精度的完美平衡。*

### 方案二：INT8 量化感知
尝试使用 RKNN Toolkit 2 对模型进行 INT8 量化（提供真实的音频特征作为校准集）。NPU 的 INT8 累加器通常使用 INT32，有时能避开 FP16 激活函数的特定 LUT 误差陷阱。但此方案需要验证量化本身是否会引入新的精度损失。

### 方案三：业务层状态重置
采用之前验证过的 `reset_interval` 策略，每隔 N 个窗口（如 1 秒）强制重置 LSTM 状态。虽然会轻微改变模型的长时依赖语义，但在某些对连续性要求不高的场景下可以作为一种工程妥协。
